DECODING WEB ATTACKS: VULNERABILITY VS. MITIGATION

SQL Injection (SQLi)

A vulnerability allowing attackers to interfere with database queries, often leading to data theft or authentication bypass. Defense relies on **Parameterized Queries**.

Vulnerable Example (PHP Pseudo-Code)

$query = "SELECT * FROM users WHERE user = '" . $input . "';"

Secure Example (PHP PDO Pseudo-Code)

$stmt = $pdo->prepare('SELECT * FROM users WHERE user = ?');
$stmt->execute([$input]);

Blind SQL Injection & Bypass

This attack occurs when the server suppresses detailed error messages. Attackers use a universal truth (e.g., **'1'='1'**) combined with the **SQL comment** (`--`) to skip the rest of the query, bypassing the password check.

Authentication Bypass Input Example

The input closes the quote, injects a universal truth, and comments out the rest of the original query.

Username: admin' OR '1'='1' -- 
Password: (any value is ignored)

Cross-Site Scripting (XSS)

Injecting malicious client-side scripts (JavaScript) into a website. Defense is **Output Encoding** and a robust **Content Security Policy (CSP)**.

Vulnerable Input

<script>alert('XSS Attack')</script>

Secure Output (Encoded)

Characters like < and > are converted to their HTML entity equivalent, rendering the script harmless.

&lt;script&gt;alert('XSS Attack')&lt;/script&gt;

Ethical Hacking Tools & Reconnaissance

Effective defense requires understanding the attacker's methods. Reconnaissance is the first step, gathering information to identify vulnerabilities.

  • **OWASP Top 10:** The global standard list of the most critical web application security risks.
  • **BURP Suite:** Works as an **interceptor** to capture and modify HTTP requests/responses. Crucial for finding **Logic Flaws**.
  • **Nmap:** A powerful network scanner used to discover live hosts and identify **open ports** and the **service versions** running on a target IP.
  • **Wappalyzer/whatweb:** Tools used in recon to identify the website's **technology stack** (e.g., WordPress, NGINX version).

NETWORK PENETRATION TESTING METHODOLOGY

A structured approach to vulnerability assessment, ensuring all vectors are considered and security controls are validated.

1

Reconnaissance (Information Gathering)

Scanning the local network (e.g., using arp-scan -l) to get the target's IP, or using tools like **Wappalyzer** to identify the website's technology stack. This step gathers initial data for the attack.

Defense: Minimize public-facing information about your infrastructure. Keep services patched to mask version details.

2

Scanning & Enumeration

Using **Nmap** to scan the target IP for services, identifying **open ports**, and detailing the software versions (e.g., nmap <target_ip> -p__,___ -sV). The real vulnerability often lies in an **outdated service version**.

Defense: Maintain rigorous patch management. Close all unnecessary ports (default ports like **RDP/3389** are common targets).

3

Exploitation (Initial Breach)

Attempting to breach the system using identified vulnerabilities, often using exploit frameworks like **Metasploit** (e.g., exploit/windows/smb/ms17_010_eternalblue) to deliver a payload (like a reverse shell).

Defense: Implement **Web Application Firewalls (WAFs)** and **Intrusion Detection Systems (IDS)** to block known exploit patterns.

4

Post-Exploitation & Reporting

Assessing the impact, maintaining access (persistence), and pivoting to other systems within the network. Finally, documenting all findings and remediation steps in a formal report.

Defense: Implement strict **Principle of Least Privilege (PoLP)** and network segmentation to limit horizontal movement after a breach.

NETWORK DEFENSE, PROTOCOLS & SECURE ACCESS

Transport & Access Protocols

Protocols define the rules of communication. Every protocol goes through a specific port.

  • **TCP (Transfer Control Protocol):** Focuses on **reliability** and **error control**. Stops transmission to retrieve lost packets.
  • **UDP (User Datagram Protocol):** Focuses on **speed** and **lightweight** communication. Continues transmission even if packets are lost (used for video streaming, DNS).
  • **QUIC:** Used in HTTP/3, it combines the speed of UDP with TCP's reliability features, reducing latency.
  • **RDP (Remote Desktop Protocol):** Allows remote access to Windows desktop. Port **3389**. Often targeted by attackers for brute-force/ransomware campaigns.
  • **Total Ports:** There are effectively **131,070** total ports (65535 * 2) because every port can use either TCP or UDP.

Secure Communication: TLS & HSTS

Secure protocols provide **encryption**, **authentication**, and **data integrity**.

  • **TLS (Transport Layer Security):** The modern successor to SSL, offering superior encryption (TLS 1.3 is the latest).
  • **HSTS (HTTP Strict Transport Security):** A policy that forces the browser to **only** connect via **HTTPS**, preventing clients from accidentally using insecure HTTP pages on a site.

TLS Handshake Overview

The initial process to establish a secure, encrypted connection:

  1. Client Hello (versions, cipher suites, random number).
  2. Server Hello (server picks options, sends **Digital Certificate**).
  3. Key Exchange & Verification (Client verifies the server's certificate).
  4. Session Key Generation (using a fast **Symmetric Key**).
  5. Finished Messages (secure communication is established).

VPN & Wireless Defense

Creating tunnels and boundaries for secure access.

  • **VPN (Virtual Private Network):** Creates a secure, **encrypted tunnel** between your device and a remote server, allowing secure remote access to a private network (e.g., using an **.ovpn client file**).
  • **WPA3 (Wi-Fi Protected Access 3):** The current mandatory Wi-Fi standard. Offers stronger encryption and **forward secrecy** (compromising one session key doesn't decrypt past data).
  • **WPA2 Vulnerability:** WPA2 is vulnerable to **KRACK attacks** and lacks protection against brute-force attacks on weak passwords.
  • **Traffic Interception:** Tools like **Wireshark** can analyze unencrypted wireless traffic, making strong encryption paramount.

THE DOMAIN NAME SYSTEM (DNS) RESOLUTION PROCESS

DNS converts a human-readable domain name (e.g., example.com) into a machine-readable IP address. This complex process involves four different servers.

Query: Client needs to resolve www.example.com.

1. Client/Local Cache

Initiates DNS query. Checks local **Cache** first for existing IP.

2. Recursive Resolver (ISP)

Acts as an intermediary. Forwards the request to the Root Nameserver if not found in cache.

3. Root Nameserver (.)

Reads the TLD (e.g., .com) and directs the query to the appropriate TLD Nameserver.

4. TLD Nameserver (.com)

Responds with the location (IP) of the domain's Authoritative Nameserver.

5. Authoritative Nameserver

The final server. Holds the actual IP address for the requested domain and sends it back to the Resolver.

Result: IP is cached locally, which is why websites load instantly on subsequent visits.

FOUNDATIONAL SECURITY: CRYPTOGRAPHY & KEY MANAGEMENT

**Cryptography** is the conversion of **plaintext** (readable) into **ciphertext** (unreadable) using an **encryption algorithm** and a **key** to ensure confidentiality, integrity, and non-repudiation.

I. Core Secure Communication Flow

  1. **Sender** prepares the Plaintext message.
  2. **Key Exchange** to agree on a secret key (either symmetric or asymmetric).
  3. **Authentication** (verify sender).
  4. **Encryption** of the plaintext into ciphertext.
  5. **Ciphertext Transmission**.
  6. **Authentication** (receiver verifies sender).
  7. **Tamper Check** (Integrity) using hashing.
  8. **Decryption** back into plaintext.
  9. **Receiver** reads the original message.

A **hybrid approach** is often used: a fast **symmetric key** (like AES) is used for bulk data, and that key is then securely exchanged using **asymmetric encryption** (like RSA).

II. Symmetric Key Encryption (Speed and Bulk Data)

Uses the **same secret key** for both encryption and decryption. This method is exceptionally fast and efficient for bulk data transfer.

Algorithm Key Length Block Size Notes
**DES** (Data Encryption Standard) 56 bits 64 bits Older standard, vulnerable to **brute-force attacks** due to short key length. Uses Feistel structure.
**AES** (Advanced Encryption Standard) 128, 192, or 256 bits 128 bits Modern standard developed to replace DES. Uses Substitution-Permutation Network. Very fast, highly secure, and used widely (WPA2/3, VPNs).

III. Asymmetric Key Encryption (Public-Key for Exchange)

Uses a mathematically linked **Public Key** (for encryption/verification) and **Private Key** (for decryption/signing). This is slower than symmetric encryption.

  • **Public Key:** Freely distributed, used to encrypt data.
  • **Private Key:** Kept secret, used only by the owner to decrypt data.
  • **Key Pairing:** Only the **Public Key** can encrypt, and only the corresponding **Private Key** can decrypt.

IV. Certificate Authorities (CAs) and Trust

CAs issue **Digital Certificates** to verify the identity of a server, binding its identity to its public key, which establishes trust in the asymmetric exchange process (like the TLS handshake).

  • **CA Role:** Verify the server's identity, sign the certificate using the CA's trusted private key, and issue the certificate.
  • **Browser Check:** The browser verifies the CA's signature using the CA's public key (already trusted by the browser). If valid, the server's public key is trusted.
  • **Well-known CAs:** DigiCert, Let's Encrypt, GlobalSign.

V. Hashing and Digital Signatures (Integrity & Non-Repudiation)

Hashing (Integrity)

A **one-way function** that produces a fixed-size **hash digest** from any input data. It ensures data **integrity** because a tiny change in the data results in a massive change in the hash, signaling tampering.

  • **Examples:** SHA-256 (for integrity), bcrypt/Argon2 (for secure password storage).

Digital Signatures

Provides **authentication** (sender verification) and **non-repudiation** (sender cannot deny sending the message).

  1. Sender calculates a **hash** of the message.
  2. Sender **encrypts** the hash using their **Private Key** (this forms the signature).
  3. Receiver calculates the hash of the received message.
  4. Receiver uses Sender's **Public Key** to **decrypt** the signature.
  5. If the two hashes match, the signature is valid.
